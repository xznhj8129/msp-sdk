# MSP SDK Implementation Plan

## Current State

The SDK skeleton exists with:
- `include/` populated with generated headers and JSON specs (from INAV)
- `generator/` has working scripts: `gen_all.sh`, `gen_msp_header.py`, `get_all_inav_enums_h.py`, `get_inav_defines.py`
- `clib/`, `arduinolib/`, `pythonlib/` are empty placeholders
- `build_c.sh`, `build_py.py` are empty stubs
- Two proof-of-concept external codebases to absorb: **MSP-Arduino** and **mspapi2**

The generator pipeline partially works: `gen_msp_header.py` generates `msp_messages.h` from `msp_messages.json`, and `get_all_inav_enums_h.py` generates `all_enums.h` from INAV source.

The C/C++ path is compile-time code generation (JSON â†’ Python script â†’ .h headers). **No runtime JSON loading in C.** The SDK produces **headers only** for C/C++/Arduino -- no codec, no runtime library, just struct definitions and protocol constants. Only Python uses runtime JSON (`mspcodec.py` loads `msp_messages.json` dynamically).

### Decided

- **MSP-Arduino:** absorb into this repo, adapt to use generated headers
- **mspapi2:** absorb into `pythonlib/`, pip-installable, loads SDK-provided JSON
- **No C codec library.** The C/Arduino output is honest headers only (structs, defines, enums). Framing/transport stays in MSP-Arduino's own code.
- **Generated files committed to git.** This is for developers, not end-users.
- **INAV version pinning:** deferred until MSP versioning is sorted

### 0.3 Header Architecture (what the consumer sees)

The end goal: a user does `#include "msp.h"` in Arduino or C and gets everything they need. Same headers work for both -- no separate C vs Arduino versions.

**What `msp_messages.h` actually depends on (4 categories):**

| Category | Examples | Source | How to get |
|----------|----------|--------|------------|
| **A. Protocol IDs** | `MSP_PROTOCOL_VERSION`, `MSP_STATUS`, `MSP2_INAV_STATUS` | `msp_protocol*.h` | Direct copy from INAV `src/main/msp/` |
| **B. Enum types** | `navMcAltHoldThrottle_e`, `gpsFixType_e`, ~60 distinct enums used as field types | `all_enums.h` | Generated by `get_all_inav_enums_h.py` from INAV source |
| **C. Array size defines** | `MAX_SUPPORTED_SERVOS`, `BUILD_DATE_LENGTH`, `OSD_CHAR_BYTES`, ~20 defines | Scattered in INAV source | Extracted by `make_min_header.py` / `get_inav_defines.py` |
| **D. Composite type stubs** | `escSensorData_t`, `ledConfig_t`, `boxBitmask_t`, `Varies`, `mixerPreset_e` | INAV source, but their real defs pull in the entire FC | Hand-written stubs with `#ifndef` guards |

**Proposed header structure:**
```

  msp.h  (umbrella, just #includes the others)
    â”śâ”€â”€ msp_protocol.h            [A] message IDs (copied from INAV)

    â”‚     Contains:
    â”‚     â”śâ”€â”€ all enums used by MSP messages  [B]
    â”‚     â”śâ”€â”€ all #define constants for array sizes  [C]
    â”‚     â””â”€â”€ composite type stubs  [D]
```

**Key decisions:**
- `msp_defs.h` replaces BOTH `msp_types.h` AND `all_enums.h` as a single generated dependency file
- **Or:** keep `all_enums.h` separate (it's already clean and generated) and have `msp_defs.h` contain only [C]+[D]. Either way works, but one file is simpler for consumers.
- `msp_stub_types.h` goes away entirely
- `msp.h` is a trivial umbrella header -- one `#include` and you're done

### 1.2 Adapt MSP-Arduino to use generated headers
Current MSP-Arduino has its own hardcoded struct definitions baked into `msp_interface.h` (e.g. `MSP_NAV_STATUS_reply_t`, `MSP_WP_reply_t`, `MSP_SET_WP_request_t`, etc.). These must be replaced with `#include "msp.h"`.

**Key changes to `msp_interface.h`/`.cpp`:**
- Remove all inline MSP struct typedefs -- use `msp_messages.h` instead
- Remove inline `#define` constants (MSP codes) -- use `msp_protocol.h` and `msp_protocol_v2_*.h`
- Keep the MSP V2 framing/CRC code as-is (it's Arduino-specific Stream-based I/O)
- Keep the high-level request/command methods as-is

**Key changes to `inav_api.h`/`.cpp`:**
- Should work mostly unchanged; it wraps `MSPIntf` methods
- Verify that struct field names in `msp_messages.h` match what `inav_api.cpp` accesses
- If field names differ, update `inav_api.cpp` to match the canonical (generated) field names

### 1.3 Implement `build_arduino.sh`
- Copies generated headers from `include/` into `arduinolib/src/`
- Copies adapted MSPInterface/InavAPI sources
- Generates `library.json` with version

## Phase 3: Build System

### 3.1 Top-level `build.sh` (final version)
```bash
#!/bin/bash
set -e
INAV_DIR=${INAV_DIR:-../inav}

# Step 1: Pull fresh sources from INAV
./fetch_inav.sh "$INAV_DIR"

# Step 2: Generate headers
./build_c.sh

# Step 3: Package Arduino library
./build_arduino.sh

# Step 4: Package Python library
./build_py.sh

echo "Build complete."
```

### 3.2 Create `fetch_inav.sh`
- Copies protocol headers from `$INAV_DIR/src/main/msp/`
- Runs enum/define extraction from INAV source
- Copies JSON specs from `$INAV_DIR/docs/development/msp/`
- Records git describe / commit hash

### 3.3 Versioning strategy
- Track INAV version the SDK was generated against in `include/VERSION`
- SDK has its own semver in `VERSION` (root)
- Generated files carry a header comment with both versions + generation timestamp

## Phase 4: Reconcile Struct/Field Naming

This is a critical integration concern: the two codebases may use different names for the same MSP fields.

### 4.1 Canonical naming: `msp_messages.json` is the source of truth
- `gen_msp_header.py` uses field names exactly as they appear in JSON
- `mspcodec.py` (Python) uses field names exactly as they appear in JSON
- MSP-Arduino uses its own hardcoded names (may differ)

### 4.2 Audit MSP-Arduino struct field names vs generated names
- Compare `MSP_NAV_STATUS_reply_t`, `MSP_WP_reply_t`, `MSP_SET_WP_request_t` field names in `msp_interface.h` against `msp_messages.h`
- List all discrepancies
- Update `inav_api.cpp` to use the canonical (generated) field names
- If JSON names are terrible, fix them in `msp_messages.json` upstream (INAV docs)

## Suggested Implementation Order

1. **Phase 0** (generator stabilization) -- prerequisite for everything
2. **Phase 4** (naming reconciliation) -- do early to avoid rework
3. **Phase 1** (Arduino lib) -- adapt MSP-Arduino to generated headers
4. **Phase 2** (Python lib) -- absorb mspapi2, mostly independent
5. **Phase 3** (build system) -- iterative, start skeleton in Phase 0
6. **Phase 5** (testing) -- iterative, add tests as each phase completes
